1. WSS 安全连接
   1. 打开网站时，客户端通过 WSS（WebSocket Secure）与服务器建立安全连接，其安全性依赖证书认证机构（CA）体系。
   2. 本毕业设计（ВКР）中，客户端与服务器的安全通信使用 Let’s Encrypt SSL 证书，该证书体系拥有全球统一的信任链，能够抵御中间人攻击（MITM）。
   3. 在实际生产环境中，为避免境外证书机构可能带来的中间人攻击风险，建议使用俄罗斯官方信任根证书（Russian Trusted Root CA）。
   4. 当安全连接成功建立后，“服务器状态” 将以绿色显示。

2. 创建新用户
   1. 用户输入密码，该密码作为对称密钥，使用 AES-GCM-256 算法加密 ECDSA 与 ECDH 私钥，并存储在浏览器的 localStorage 中，防止未授权访问。
   2. 点击 “创建” 按钮后，客户端向服务器请求用户 ID。
   3. 服务器生成并返回用户 ID 给客户端。
   4. 客户端生成 ECDSA 和 ECDH 密钥对，使用 ECDSA 私钥对用户 ID 进行签名，并将 ECDSA 公钥、ECDH 公钥及签名发送给服务器。
   5. 服务器使用客户端公钥验证签名，验证通过后保存用户 ID、ECDSA 公钥和 ECDH 公钥。
   6. 客户端收到验证成功消息后，将加密后的私钥与公钥存入 localStorage，供后续使用。
   7. 在实际生产环境中，应使用俄罗斯国产 ГОСТ 密码算法：对称加密采用 Кузнечик 替代 AES-GCM，椭圆曲线数字签名与密钥交换采用 ГОСТ Р 34.10-2012/2018 替代 ECDSA 和 ECDH。

3. 加载用户与在线用户列表
   1. 从本地 localStorage 中加载已保存的用户信息。
   2. 用户输入密码后，程序尝试解密 ECDSA 和 ECDH 私钥；解密成功后，使用 ECDSA 私钥对用户 ID 签名，并将签名与用户 ID 发送至服务器。
   3. 服务器根据用户 ID 获取已保存的 ECDSA 公钥，验证签名有效性；验证通过后将用户加入在线列表，并返回认证结果。
   4. 客户端收到验证通过结果后，在界面上以绿色显示用户 ID。
   5. 用户 ID 加载成功后，可从服务器获取当前在线用户列表。
   6. 选择目标用户后，进入第四步，发起 WebRTC 点对点连接。

4. WebRTC 连接建立
   1. WebRTC 连接发起方（点击 “连接” 按钮）称为 Caller，接收方称为 Callee；offer 和 answer 是建立 WebRTC 连接必需的协商信息。
   2. Caller 创建 RTC 点对点连接，生成 offer，使用 ECDSA 私钥签名自身用户 ID，并将签名与 offer 发送至服务器。
   3. 服务器通过 WSS 将 Caller 的签名、ECDSA 公钥、ECDH 公钥及 offer 转发给 Callee。
   4. Callee 根据 offer 创建连接并生成 answer，使用 Caller 公钥验证其签名，再用自身 ECDSA 私钥签名用户 ID，并将签名与 answer 发回服务器。
   5. 服务器通过 WSS 将 Callee 的签名、ECDSA 公钥、ECDH 公钥及 answer 转发给 Caller。
   6. Caller 完成 WebRTC 连接建立，并使用 Callee 公钥验证其身份签名。
   7. 该过程通过 WSS 传输公钥、offer 和 answer，可防止第三方中间人攻击，但无法防范服务器自身作恶。

5. OOB 带外身份验证（核心安全机制）
   1. 如第四步所述，WSS 仅能防范外部中间人攻击，无法验证服务器本身是否可信。互联网无法构建绝对安全的信任体系，任何被信任的节点都可能存在背叛风险；服务器可作为中间人实施攻击，而其依赖的 CA 体系也可能被美国利用。
   2. 开源代码无法完全解决该问题。前端 HTML 是公开的，可通过 SHA 哈希校验页面完整性，防止服务端注入恶意 JS 窃取私钥；但无法保证服务器实际运行代码与开源代码完全一致。
   3. 自建服务器可避免服务器中间人攻击，自签名证书可摆脱对美国主导 CA 体系的依赖，但对普通用户技术门槛过高，部署成本显著上升。
   4. WebRTC 用于实现点对点通信、降低服务器压力；ECDH 用于实现端到端加密，保证通信机密性。当通信双方均处于对称型 NAT 时，无法通过 STUN 建立直连，必须通过 TURN 服务器中转流量。尽管 WebRTC 自身使用 DTLS+SRTP 加密，但服务器作恶时仍可能实施中间人攻击。
   5. 因此，本设计不依赖 WebRTC、CA 体系或服务器信任来保证消息安全，而是采用带外身份验证（OOB Authentication） 验证双方身份。主流端到端加密软件（WhatsApp、Telegram、Signal）均采用该机制：例如 Telegram 会显示四个表情，双方表情一致则通信安全。
   6. WebRTC 连接过程中，服务器向双方转发对方的 ECDSA 和 ECDH 公钥。客户端通过 ECDSA 公钥验证签名后，在用户 ID 后显示✅；随后对双方公钥计算 SHA-256 指纹，并根据指纹生成可视化表情以便人工比对。
   7. 通信双方需在线下或已信任通道中验证指纹一致性，验证通过后方可安全通信；同时独立校验页面 MD5 是否与开源版本一致。这是唯一需要信任的两个环节，即使其余信道不可信，仍能保证通信安全。
   8. 在实际生产中，应使用 Стрибог（ГОСТ Р 34.11-2012）哈希算法替代 SHA-256。

6. 明文消息区域
   1. 此处显示 WebRTC 通信中的原始明文，包括本区域发送的明文消息，以及下方区域发送的密文原始格式。

7. 加密消息区域
   1. 此处显示解密后的加密消息。程序会自动判断数据是否为 JSON 格式，并检查 type 字段是否为 encryptedMessage；符合条件的密文将被解密并展示在该区域。
